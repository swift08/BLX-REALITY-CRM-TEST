# BLX Realty CRM — Deployment Guide

> **Audience**: Deployment engineers, Vercel admins, and the BLX Realty technical team.
> **Last updated**: July 2026

---

## Prerequisites

- Node.js 18+ (or Bun 1.x)
- A Supabase project (PostgreSQL + Auth enabled)
- A Vercel account connected to the repository
- (Optional) A Meta Developer App with `leads_retrieval` permission

---

## Step 1 — Configure Environment Variables

### On Vercel Dashboard

Go to: **Project → Settings → Environment Variables**

Add all variables from `.env.example`. Below is the full variable reference:

| Variable                    | Where Used               | Required                    |
| --------------------------- | ------------------------ | --------------------------- |
| `VITE_SUPABASE_URL`         | Client (browser)         | ✅ Yes                      |
| `VITE_SUPABASE_ANON_KEY`    | Client (browser)         | ✅ Yes                      |
| `SUPABASE_URL`              | Server (API functions)   | ✅ Yes                      |
| `SUPABASE_SERVICE_ROLE_KEY` | Server (API functions)   | ✅ Yes                      |
| `ALLOWED_ORIGIN`            | CORS origin lock         | ✅ Yes (set to your domain) |
| `META_APP_ID`               | Meta Webhook             | ⚠️ If using Meta Lead Ads   |
| `META_APP_SECRET`           | Meta Webhook (HMAC sig)  | ⚠️ If using Meta Lead Ads   |
| `META_VERIFY_TOKEN`         | Meta Webhook handshake   | ⚠️ If using Meta Lead Ads   |
| `META_ACCESS_TOKEN`         | Graph API lead retrieval | ⚠️ If using Meta Lead Ads   |
| `META_GRAPH_VERSION`        | Graph API version        | Optional, defaults to v23.0 |

> ⚠️ **CRITICAL**: `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are **NOT** the same as `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`. They are separate variables for the server-side API. Both sets must be configured or the application will fail at runtime.

> ⚠️ **SECURITY**: Never use the service role key in client-side code. Only server API functions (`api/crm.ts`, `api/meta/webhook.ts`) should ever read `SUPABASE_SERVICE_ROLE_KEY`.

## Step 2 — Run Supabase Schema Migrations

Open your Supabase project → **SQL Editor** and run `supabase_schema.sql` in full.

The schema is idempotent (`CREATE TABLE IF NOT EXISTS`, `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`) — it is safe to re-run without data loss.

**Recommended order if running incrementally:**

1. Tables: developers, projects, customers, opportunities, inventory
2. Tables: bookings, invoices, payments, interactions, documents
3. Tables: followups, calendar_events, workflow_rules, audit_logs
4. Tables: notifications, notes, settings, invoice_settings
5. Tables: meta_webhook_logs, lead_assignment_settings, lead_assignment_history
6. Tables: negotiation_details, negotiation_timeline
7. **Migration blocks**: All `ALTER TABLE` statements at bottom of file (bookings + invoices columns)

After running, confirm schema is visible:

```sql
SELECT table_name FROM information_schema.tables WHERE table_schema = 'public';
```

Reload the PostgREST schema:

```sql
NOTIFY pgrst, 'reload schema';
```

## Step 3 — Configure Supabase Auth

1. In Supabase → **Authentication → Settings**, set **Site URL** to your deployment domain.
2. Add your domain to **Redirect URLs** (e.g. `https://crm.blxrealty.com/**`).
3. Disable **email confirmations** if you want invite-based user creation without confirmation flow.
4. Set a strong **JWT secret** (auto-generated by Supabase — do not change unless you know the implications).

### Creating the First Super Admin

Use the Supabase Dashboard or SQL to create the first user, then set their metadata:

```sql
UPDATE auth.users
SET raw_user_meta_data = jsonb_set(
  raw_user_meta_data,
  '{role}',
  '"super_admin"'
)
WHERE email = 'your-admin@email.com';
```

## Step 4 — Deploy to Vercel

```bash
# From the project root
npm run build   # Verify clean build locally first
git push        # Vercel auto-deploys on push to main
```

Vercel will:

1. Run `vite build` to produce `dist/`
2. Serve `dist/` as a static SPA
3. Serve `api/crm.ts` as a serverless function at `/api/crm`
4. Serve `api/meta/webhook.ts` as a serverless function at `/api/meta/webhook`

The `vercel.json` rewrites route all `/(.*)` → `index.html` for SPA routing, and `/api/(.*)` → serverless functions.

## Step 5 — Meta Lead Ads Webhook (Conditional)

> ⚠️ **This section only applies if you are using Facebook/Instagram Lead Ads integration.**

### Current Status

The webhook endpoint (`/api/meta/webhook`) is **technically complete**:

- HMAC-SHA256 signature verification ✅
- Idempotency guard via `meta_webhook_logs` ✅
- PII-safe production logging (no lead data in server logs) ✅
- Automatic lead creation in CRM ✅
- Duplicate detection via `meta_lead_id` ✅

### Meta App Review Requirement

⚠️ **Action Required**: The `leads_retrieval` permission for Facebook Lead Ads **requires Meta App Review approval before production leads will be delivered.**

Without App Review approval:

- The webhook will work in test mode (only test leads from the Meta Developer Dashboard)
- Real customer leads from live Facebook/Instagram campaigns will NOT be delivered

**What you need to do:**

1. Log into [developers.facebook.com](https://developers.facebook.com)
2. Go to your App → App Review → Permissions & Features
3. Request `leads_retrieval` permission
4. Submit with screenshots of the CRM webhook flow for Meta's review team

**Current App Review Status**: ⬜ Not yet submitted / 🟡 Under review / ✅ Approved  
_(Update this field when status changes)_

### Webhook Registration

Once Meta App Review is approved:

1. Go to your Meta App → Webhooks
2. Subscribe to Page: `leadgen` field
3. Callback URL: `https://your-domain.com/api/meta/webhook`
4. Verify token: `META_VERIFY_TOKEN` (from your .env)

## Smoke Test Checklist

Run these checks after every deployment before handing off to a client:

- [ ] Login with Super Admin credentials → lands on Dashboard
- [ ] Create a test customer → appears in Leads list
- [ ] Assign the customer to a Sales Executive → assignment visible in Lead detail
- [ ] Create a follow-up → visible in Followups route
- [ ] Mark follow-up complete → appears as completed in the list
- [ ] Schedule and complete a site visit → stage advances to `site_visit_completed`
- [ ] Transition lead to Negotiation → Negotiation Workspace tab appears
- [ ] Create a booking → shows in Bookings route
- [ ] Generate an invoice → PDF renders correctly
- [ ] Navigate to all routes: Dashboard, Projects, Inventory, Calendar, Analytics, Settings, Users
- [ ] Log out and confirm session is cleared

## Rollback Procedure

1. In Vercel Dashboard → **Deployments**, find the last good deployment
2. Click **Promote to Production** on that deployment
3. No database rollback needed (all schema changes are additive — no destructive DDL)

---

## Security Notes

- **Service role key is never client-side.** The browser only holds the Supabase Anon Key. All privileged database ops go through `api/crm.ts` which verifies the JWT before each action.
- **CORS is locked** to `ALLOWED_ORIGIN` in production. Set this to your exact domain.
- **Audit logs are immutable** — no UI action allows editing or deleting audit log entries.
- **PII in production logs is suppressed** — webhook logs are gated behind `NODE_ENV !== 'production'`.
